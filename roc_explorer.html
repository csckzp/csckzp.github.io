Understanding ROC Curves

ROC (Receiver Operating Characteristic) curves help evaluate binary classification models by plotting True Positive Rate vs False Positive Rate across different thresholds.

Key Concepts:

• True Positive Rate (TPR) = Sensitivity = TP / (TP + FN) - How many actual positives we correctly identified

• False Positive Rate (FPR) = 1 - Specificity = FP / (FP + TN) - How many actual negatives we incorrectly classified as positive

• AUC (Area Under Curve) - Overall model performance. Higher is better (0.5 = random, 1.0 = perfect)

How to Use: Move the threshold slider to see how different cutoff points affect the trade-off between catching true positives and avoiding false positives. The shaded area represents the AUC score.

📚 ROC & AUC, Quick Primer

A classifier assigns every instance a score from 0 to 1. Picking a threshold turns those scores into "positive" / "negative" predictions. The ROC curve plots True Positive Rate (TPR) against False Positive Rate (FPR) as that threshold sweeps from 1 down to 0, showing every possible trade-off a single threshold choice locks you into.

  • TPR (Sensitivity/Recall) = TP / (TP + FN) — fraction of real positives you catch.
  • FPR = FP / (FP + TN) — fraction of real negatives you wrongly flag.
  • AUC = area under that curve — a threshold-free summary of separability. 0.5 is random guessing, 1.0 is a perfect ranking of positives above negatives.

🕹️ Playing with the Demo

  • Drag the Classification Threshold slider — anything scored at or above it is predicted positive.
  • Watch the red dot slide along the ROC curve and the dashed line move through the Score Distribution histogram in lockstep.
  • Open the Ranked Instances tab to see exactly which individual points became TP/FP/FN/TN at the current threshold.
  • AUC stays fixed as you drag — it's a property of the scores, not of any one threshold. Only TPR, FPR, and Accuracy change.

🔧 Settings Worth Trying

  • Threshold ≈ 1.0: almost nothing is predicted positive — FPR near 0, but TPR (recall) near 0 too. Watch the False Negatives column fill up.
  • Threshold ≈ 0.0: almost everything is predicted positive — TPR near 1, but FPR near 1 too. Watch False Positives dominate.
  • Threshold ≈ 0.5: the balanced default — compare its Accuracy to the extremes above.
  • Find the threshold that hugs the ROC curve's top-left corner — that's usually the best joint trade-off between TPR and FPR.
  • Compare the curve's bow against the gray diagonal: the more it bulges toward the top-left, the higher the AUC and the better the model separates the two classes.

⚖️ When to Minimize False Positives vs False Negatives

You can't usually minimize both at once — lowering one raises the other. Which one to prioritize depends on which mistake is more costly:

🔵 Minimize False Positives (raise the threshold)

Use when flagging a negative as positive is the expensive/harmful mistake:

  • Spam filtering — a false positive buries a real, possibly important email.
  • Criminal justice / fraud accusations — wrongly flagging an innocent person or legitimate transaction has serious consequences.
  • Marketing / promo eligibility — giving a discount to someone who wasn't going to churn wastes margin.
  • Content moderation (auto-removal) — wrongly taking down legitimate content harms users and trust.
  • Automated approval systems (e.g., loan pre-approval) — approving a bad candidate outright is costlier than sending a good one to manual review.

🔴 Minimize False Negatives (lower the threshold)

Use when missing a true positive is the expensive/harmful mistake:

  • Medical screening (cancer, disease tests) — missing an actual illness can be life-threatening; a false alarm just means a follow-up test.
  • Fraud / intrusion detection — missing real fraud or a breach can cause major financial or security damage.
  • Airport / security screening — missing an actual threat is far worse than an extra bag check.
  • Predictive maintenance — failing to flag a part that will actually fail can cause equipment damage or safety incidents.
  • Search & rescue / disaster alerts — missing a real emergency signal is far costlier than a false alarm.

Tip: come back to this demo and set the threshold near the extremes to see, in the Ranked Instances tab, exactly which real cases you'd be trading off in each scenario.
